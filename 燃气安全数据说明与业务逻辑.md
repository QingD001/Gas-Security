# 燃气安全数据说明与业务逻辑

本文只解释数据含义和业务逻辑，不讨论模型效果。

## 一、整体关系

当前项目里有两条业务线，容易混在一起。

第一条是**误报识别**：

```text
报警已经发生
系统或人工已经记录了报警事件
模型判断这次报警是不是误报
```

这条线需要人工核实标签。旧文件 `data.csv` 属于这类数据。

第二条是**提前预警**：

```text
当前还没有报警
系统持续收到设备监测值
模型判断未来短时间内是否可能报警
```

这条线不依赖人工误报标签，而是用设备阈值表定义“是否报警”。新接入的 9GB 原始流水和阈值表属于这条线。

两条线的目标不一样，不能直接把数据混在一起训练。

## 二、原始监测流水表

文件：

```text
rq_jc_ssjc_202605131842.csv
data/rq_jc_ssjc_part_001.csv ~ data/rq_jc_ssjc_part_071.csv
```

含义：

```text
一行 = 某个设备在某个时间上报的一次监测值
```

它不是报警表，也不是人工标注表。

主要字段：

| 字段 | 含义 |
| --- | --- |
| `bsm` | 当前流水记录的唯一标识 |
| `sbbsm` | 设备标识码 |
| `dwbsm` | 单位或点位标识码 |
| `jczb` | 监测指标名称，例如可燃气体浓度、压力、液位、温度 |
| `jcz` | 监测值 |
| `jccjsj` | 采集时间 |
| `jcsbsj` | 上报时间 |
| `bz` | 备注，通常包含点位名称和指标名称 |
| `sbzt` | 设备状态 |
| `yskzjbz` | 原始扩展标记 |
| `sjtbzt` | 数据填报状态 |
| `tbsj` | 填报或入库时间 |
| `jczdw` | 监测单位，目前基本为空 |

这张表：

```text
某设备在某时刻的监测值是多少？
这个设备近期监测值怎么变化？
是否接近报警阈值？
```


## 三、设备阈值规则表

文件：

```text
sbyxwh_jcxyz_202605131521.csv
```

含义：

```text
一行 = 某个设备某个指标的一条报警阈值规则
```

主要字段：

| 字段 | 含义 |
| --- | --- |
| `bsm` | 阈值规则记录 ID |
| `sbbsm` | 设备标识码 |
| `jczb` | 监测指标编码，例如 `RQ0101`、`PS0101` |
| `bjyz` | 报警阈值 |
| `pdgz` | 判断规则类型 |
| `bjjb` | 报警等级 |
| `sjlx` | 数据类型 |
| `create_time` | 规则创建时间 |
| `update_time` | 规则更新时间 |
| `delete_flag` | 删除标记 |
| `yzsx` | 阈值上限，当前大量为空 |
| `yzxx` | 阈值下限，当前大量为空 |

业务上，它定义的是：

```text
某个设备的某个指标，达到什么值时算报警。
```

例如：

```text
sbbsm = 1000000000000264827
jczb = RQ0101
bjyz = 20
bjjb = 3
```

可以理解为：

```text
设备 1000000000000264827 的 RQ0101 指标达到 20 时，触发 3 级报警规则。
```

同一个设备和指标可能有多条阈值，对应不同报警级别。当前二分类任务先用最低阈值判断“是否报警”。

## 四、指标名称与指标编码

原始流水表里的 `jczb` 是中文名称，阈值表里的 `jczb` 是编码。两张表要先做映射。

当前映射关系：

| 原始流水指标 | 阈值表指标编码 |
| --- | --- |
| 可燃气体探测器_可燃气体浓度 | `RQ0101` |
| 燃气压力设备_压力 | `PS0101` |
| 液位传感器_液位 | `RQ0401` |
| 温度传感器_温度 | `RQ0501` |

匹配键是：

```text
设备标识码 sbbsm + 指标编码 jczb_code
```

也就是说，一条监测流水要先把中文指标转成编码，再去阈值表里找该设备该指标的阈值。

## 五、当前是否报警的定义

当前项目里，“是否报警”不是模型自己定义的，也不是人工标注出来的。

它来自阈值规则：

```text
当前监测值 value >= 报警阈值 threshold
=> 当前报警

当前监测值 value < 报警阈值 threshold
=> 当前不报警
```

代码里的含义是：

```python
current_alarm = value >= threshold
```

这里：

```text
value 来自原始流水表的 jcz
threshold 来自阈值表的 bjyz
```

这套定义适合做报警/不报警，不适合做误报/非误报。

## 六、旧报警事件样本表

文件：

```text
data.csv
```

含义：

```text
一行 = 一次已经发生的报警事件
```

它已经不是原始流水，而是被整理过的报警宽表。

主要字段：

| 字段 | 含义 |
| --- | --- |
| `报警编号` | 报警事件编号 |
| `报警等级` | 事件报警等级 |
| `现场工作人员核实-是否误报` | 现场人员核实结果 |
| `综合研判-误报` | 综合研判标签 |
| `监测指标` | 指标编码 |
| `设备标识码` | 设备标识码 |
| `报警生成时间` | 报警生成时间 |
| `初次报警时间` | 首次报警时间 |
| `初次报警值` | 首次报警值 |
| `报警阈值` | 报警阈值 |
| `报警前10次推送值` | 报警前历史值 |
| `报警后1次推送值` 等 | 报警后监测值 |

这张表适合做：

```text
误报 / 非误报识别
```

因为它有人工或业务系统整理过的误报标签。

它不适合直接和新流水表混用，因为两者粒度不同：

```text
data.csv：一行是一次报警事件
原始流水：一行是一次设备采样
```

## 七、当前构造出的训练数据

### 1. 当前报警样本

文件：

```text
alarm_dataset.csv
```

含义：

```text
一行 = 某设备某时刻的监测值，以及它当前是否达到阈值
```

标签：

```text
label = 当前 value 是否 >= threshold
```

这份数据主要用于验证链路：

```text
流水表能否匹配阈值表？
能否生成基本报警/不报警样本？
```

它不适合评估提前预警能力，因为当前值和阈值本身就决定了当前是否报警。

### 2. 未来报警样本

文件：

```text
alarm_future_dataset.csv
```

含义：

```text
一行 = 当前未报警时刻的一个预测样本
```

输入特征来自：

```text
当前值
过去 10 次监测值
过去 10 次的统计量
阈值
指标编码
```

标签来自：

```text
未来 3 次监测中是否有一次达到报警阈值
```

代码逻辑可以理解为：

```text
当前 value < threshold
并且未来 3 次里存在 future_value >= threshold
=> label = 1

当前 value < threshold
并且未来 3 次都没有达到阈值
=> label = 0
```

这是当前更严谨的提前预警数据集。

主要字段：

| 字段 | 含义 |
| --- | --- |
| `sbbsm` | 设备标识码 |
| `jczb` | 原始中文指标 |
| `jczb_code` | 指标编码 |
| `time` | 当前样本时刻 |
| `value` | 当前监测值 |
| `threshold` | 当前设备指标的报警阈值 |
| `rule_level` | 阈值规则对应报警等级 |
| `label` | 未来窗口内是否报警 |
| `current_alarm` | 当前是否已经报警，当前样本中应为 0 |
| `future_max` | 未来窗口内最大监测值 |
| `future_minutes` | 未来窗口的最近时间跨度 |
| `pre_1` ~ `pre_10` | 当前之前 10 次监测值 |
| `pre_mean` | 过去 10 次均值 |
| `pre_max` | 过去 10 次最大值 |
| `pre_min` | 过去 10 次最小值 |
| `pre_std` | 过去 10 次标准差 |
| `pre_zero_rate` | 过去 10 次中 0 值比例 |
| `diff_1` | 当前值减最近一次历史值 |
| `diff_mean` | 历史序列平均变化 |
| `value_vs_threshold` | 当前值与阈值的比例 |

## 八、三类任务不要混淆

### 任务 A：当前报警判断

问题：

```text
当前这个值是否报警？
```

答案：

```text
看阈值规则即可。
```

公式：

```text
value >= threshold
```

这类任务不需要机器学习。

### 任务 B：未来报警预测

问题：

```text
当前还没报警，未来短时间是否可能报警？
```

答案：

```text
需要时序模型或规则 + 模型。
```

当前项目主线就是这个。

可用数据：

```text
原始监测流水 + 阈值规则表
```

### 任务 C：误报识别

问题：

```text
报警已经发生，这次报警是不是误报？
```

答案：

```text
需要人工核实标签。
```

可用数据：

```text
data.csv
```

新接入的 9GB 原始流水不能直接回答误报问题。

## 九、业务流程理解

当前数据流可以理解为：

```text
设备持续上报监测值
        ↓
原始流水表记录每次采样
        ↓
根据设备 + 指标匹配阈值表
        ↓
判断当前是否达到报警阈值
        ↓
如果要做提前预警：
    用当前和历史监测值预测未来是否会达到阈值
        ↓
如果真的触发报警并进入处置流程：
    才会产生报警事件、核实结果、误报标签
```

所以，报警预测发生在报警之前；误报识别发生在报警之后。

## 十、当前项目建议

当前最清晰的推进路线是：

```text
先做未来报警预测
再视情况接入人工核实标签做误报识别
```

近期重点：

- 补充时间间隔特征，因为设备采样并不是严格等间隔。
- 调整未来预测窗口，例如未来 5 次、10 次、30 分钟、60 分钟。
- 完善指标映射，确认压力、液位、温度等指标能否稳定匹配阈值。
- 做按时间和按设备的外推测试，避免模型只记住局部历史。

一句话总结：

```text
原始流水负责描述设备怎么变化；
阈值表负责定义什么时候算报警；
旧 data.csv 负责误报识别；
新构造的 future 数据集负责提前预警。
```
