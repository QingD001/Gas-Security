# 燃气安全预警项目进度

日期：2026-05-22

这次需要先把任务口径校正一下。我们不是要做“用过去 10 次预测未来几次会不会报警”，也不是简单复现阈值规则。现在更准确的目标是：

```text
给一段连续监测流水打一个报警风险分。
```

也就是把某个设备、某个指标的一段 20 次左右的流水作为输入，模型输出这段数据“像不像真实报警”的概率。

## 1. 当前任务应该这样理解

一条样本不是单个采样点，也不是“未来几次是否超过阈值”，而是一段时序窗口。

```text
输入：某设备某指标连续 20 次监测值
输出：这 20 次流水对应的报警风险概率
```

这个概率可以和业务侧标签对比：

```text
真实报警窗口 -> 期望模型给高分
未触发报警窗口 -> 期望模型给低分
触发了但被标成误报的窗口 -> 也期望模型给低分
```

也就是说，模型目标不是判断“有没有超过阈值”，而是判断这段数据整体是否像一次有效报警。

这个口径比之前更贴近业务。因为阈值规则只能说“有没有触发报警”，但不能说明这段变化是不是可信、是不是值得处置。

## 2. 和之前几个任务的区别

### 当前值是否报警

这个只看规则：

```text
value >= threshold
```

规则能解决，不需要机器学习。

### 未来是否报警

这是我前面理解成的方向：

```text
当前未报警
用过去若干次
预测未来几次是否越过阈值
```

这个口径目前不作为主线。

### 20 次窗口报警风险评分

这才是现在要做的方向：

```text
取一段 20 次流水
不管中间是否已经触发阈值
不直接把“是否越阈值”当最终目标
而是预测这段数据整体属于真实报警的概率
```

如果一段数据触发了报警但业务标签是误报，模型应尽量给低分。

如果一段数据没有触发报警，模型也应给低分。

如果一段数据是真实报警前后对应的有效报警窗口，模型应给高分。

## 4. 样本应该怎么构造

### 正样本

从业务标注为真实报警的报警事件里取窗口。

窗口可以围绕报警时刻取：

```text
报警前 20 次
或 报警前 10 次 + 报警后 10 次
或 报警附近连续 20 次
```

这几种可以分别试。核心是正样本代表“这段流水确实对应一次有效报警”。

### 负样本

负样本至少包括两类。

第一类是没有触发报警的普通流水窗口：

```text
同设备同指标随机取连续 20 次
窗口内没有报警事件
没有明显越阈值
```

第二类是业务标注为误报的报警窗口：

```text
虽然触发过报警
但人工或综合研判认为是误报
```

这一类很重要。它能让模型学到：超过阈值不一定就是有效报警。

## 5. 模型输入特征

一段 20 次流水可以直接展开成序列特征：

```text
v_1 ~ v_20
dt_1 ~ dt_20
v_i / threshold
threshold - v_i
```

也要加统计特征：

```text
均值
最大值
最小值
标准差
零值比例
非零值持续次数
接近阈值的比例
超过阈值的次数
最大跃升幅度
单位时间变化率
窗口总时长
```

阈值不是最终答案，但可以作为辅助特征。比如同样是 10 的值，在阈值 20 和阈值 80 的设备上意义不一样。

## 6. 模型输出

模型输出不是“是否触发阈值”，而是：

```text
alarm_score = 这 20 次流水像真实报警的概率
```

后面可以按阈值使用：

```text
alarm_score >= 0.8  高风险，优先关注
alarm_score 0.5~0.8 中风险，进入观察
alarm_score < 0.5  低风险，可认为不像真实报警
```

这个分数可以和业务标签对齐验证。

## 7. 评价方式

评价时不能只看模型有没有预测“越阈值”。

应看三类窗口的分数分布：

```text
真实报警窗口：分数是否整体偏高
误报窗口：分数是否整体偏低
未报警窗口：分数是否整体偏低
```

更直观的评价方式：

```text
真实报警平均分
误报平均分
未报警平均分
三类分数箱线图
Top K 高分窗口里真实报警占比
在固定召回率下误报减少多少
```

如果真实报警窗口分数明显高，误报和未报警窗口分数明显低，说明模型目标达到了。

## 9. 下一步计划

### 1. 重做窗口数据集

新建一个窗口级数据集，例如：

```text
alarm_window_20.csv
```

一行代表一个 20 次流水窗口。

字段大致包括：

```text
sbbsm
jczb_code
window_start_time
window_end_time
v_1 ~ v_20
dt_1 ~ dt_20
threshold
window_label
label_type
```

其中：

```text
window_label = 1 真实报警窗口
window_label = 0 误报窗口或未报警窗口

label_type = true_alarm / false_alarm / no_alarm
```

### 2. 对齐业务标签

需要把 `data.csv` 里的报警事件和原始流水按下面字段对齐：

```text
设备标识码
监测指标
报警时间
```

对齐后取报警附近 20 次流水，作为带业务标签的窗口。

### 3. 训练窗口评分模型

先用 XGBoost 做结构化窗口特征。

如果效果不够，再考虑：

```text
LSTM
Transformer
时间间隔编码
异常检测模型
```

### 4. 重点看分数分布

组会评估时重点展示：

```text
真实报警窗口分数
误报窗口分数
未报警窗口分数
```

不是只报 accuracy。

## 10. 组会口径

本轮需要修正任务理解：我们不是要用前几次数据预测未来是否越过阈值，而是要给一段连续流水打报警风险分。

模型输入是一段 20 次左右的设备监测序列，输出这段序列像真实报警的概率。后续用这个概率和业务标签对比：真实报警应高分，误报和未报警应低分。
